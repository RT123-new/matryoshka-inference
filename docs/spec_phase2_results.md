# Spec Phase 2 Results — Token-ID Verified Speculation (and a Hardened Text Probe)

This records Phase 2 from `HANDOFF.md`: move the losslessness guarantee off
decoded token *surfaces* and character offsets and onto **target-engine token
IDs**, and harden the existing text-surface probe so it cannot classify an
incompatible endpoint as safe.

**Bottom line up front.** The token-ID verifier is **id- and byte-identical to
plain raw-argmax greedy generation** across every deterministic test — including
the exact cases text mode cannot handle (distinct ids sharing a surface;
output that re-tokenizes differently). The text-surface probe is now **strict**:
byte offsets, incomplete echo, partial coverage, ambiguous alignment, missing
bonus, malformed/non-finite logprobs and non-greedy policies are all rejected,
and `spec_generate` cannot enter the verify lane without a usable capability.
**No trained model and no real GGUF were reachable in this environment**, so
token-ID equivalence is demonstrated against deterministic fixtures and set up
(opt-in) for a real engine, not measured on one. **No speed claim is made** —
the correctness gate is the deliverable.

## 2. Model / fixture types used

Everything measured here is **synthetic and deterministic** — the honest
consequence of having no reachable trained model:

- **Text-surface lane** — `sclab.spec.sim.SimEngine` (a "lag LM": greedy token at
  position *p* is the token at *p − lag*), driven over real HTTP with exact
  `echo` + `logprobs` + `text_offset` semantics, in both the classic (shift 0)
  and llama-cpp-python (shift 1) conventions, and in **code-point** and **UTF-8
  byte** offset units.
- **Token-ID lane** — deterministic fake token backends exposing real ids and a
  raw-argmax oracle: a canonical char/merge tokenizer + lag model
  (`LagTokenBackend`), and a scripted-id model (`PlaybackBackend`) that lets a
  test place duplicate-surface ids, non-canonical merges and EOS exactly.

No synthetic result is presented as a trained-model result. The **synthetic
cycle** and **pathological random** GGUF fixtures from Phase 1, and any locally
available trained GGUF, are wired into opt-in tests (`§8`) that **skip** here.

## 3. Capability classification

### 3.1 Text-surface probe (`verify.probe_endpoint`) — now strict

An endpoint is usable **only** when *all* invariants hold at once. New
classifications (all reachable, each with an adversarial test in
`tests/test_spec_capability_strict.py`):

| classification | meaning | usable |
|---|---|---|
| `verified_text_classic_alignment` | index *i* predicts token *i* (shift 0) | ✅ (surface only) |
| `verified_text_shifted_alignment` | index *i* predicts token *i+1* (shift 1) | ✅ (surface only) |
| `echo_ignored` | prompt not echoed at all (native `llama-server`) | ❌ |
| `echo_incomplete` | scored input not echoed in full | ❌ |
| `generated_logprobs_only` | logprobs cover generated positions only | ❌ |
| `unsupported_response_shape` | missing tokens/top_logprobs/offsets | ❌ |
| `malformed_logprob_arrays` | length mismatch or non-finite logprobs | ❌ |
| `partial_continuation_coverage` | continuation not tiled by whole tokens | ❌ |
| `invalid_offsets` | non-monotonic / overlapping / gapped / non-mapping | ❌ |
| `unsupported_offset_units` | UTF-8 **byte** offsets, not code points | ❌ |
| `ambiguous_alignment` | continuation verifies at **both** shifts | ❌ |
| `bonus_unavailable` | no present, unambiguous bonus prediction | ❌ |
| `unsupported_token_identity` | byte-fallback (`<0xE2>`) surfaces | ❌ |
| `nondeterministic_policy` | response advertises a non-greedy/unknown policy | ❌ |

Two safety-relevant hardening details:

- **`usable` now requires `offsets_ok` *and* `bonus_ok` *and*
  `continuation_verified`.** A capability with either false can never be usable
  (Phase 1 let a "verified" status through with `offsets_ok=False`).
- **The probe prompt is deliberately multibyte** (`café`, `über`). On pure ASCII,
  byte and code-point offsets coincide, so a byte-offset endpoint would have
  slipped through and then mis-indexed real multibyte generation. The multibyte
  probe forces the offset unit to reveal itself.

Measured (`results/spec_phase2/capability.json`): the code-point sim →
`verified_text_classic_alignment` (usable); the **same sim with byte offsets** →
`unsupported_offset_units` (**rejected**).

### 3.2 Token-ID backend (`backend.VerificationCapability`)

`verified_token_id` (usable) requires `deterministic=True` and records the exact
raw-argmax policy enforced (temperature 0, top-k 1, top-p/min-p off, all
repetition/presence/frequency penalties 1/0, mirostat off, no grammar/logit-bias).
`policy_is_deterministic` rejects any unknown field or non-greedy value, so a new
sampling knob cannot slip through unverified. The in-process
`llama-cpp-python` backend reports `token_id_unavailable` if the model was loaded
without `logits_all=True`.

## 4. Verification basis — text-surface vs token-ID

| | Text-surface mode | Token-ID mode |
|---|---|---|
| Compares | decoded **surfaces** + char offsets | **token ids** |
| Authoritative state across rounds | text string | **token ids** (never re-tokenized) |
| Guarantee | *surface* identity, raw-argmax greedy, on a probed endpoint | **unconditional id + byte identity** |
| Distinct ids, same surface | indistinguishable | distinguished |
| Output re-tokenizes differently | can break (seam fallback) | cannot corrupt state |
| Status | **conditional, experimental** | Phase 2 core |

## 5. Correctness results — token-ID (the gate)

`spec.token_ids == baseline.token_ids` **and** `spec.text_bytes ==
baseline.text_bytes`, where the baseline is one plain `generate_plain` call. Text
equality alone is explicitly **not** the gate.

- **All 12 required token-ID cases pass** (`tests/test_spec_token_verify.py`, 42
  tests): fully-correct draft accepted; first / middle mismatch corrected; full
  draft emits the correct bonus; no token skipped; exact max-token budget; EOS
  stops at the same place; duplicate decoded surface with distinct ids; a
  non-canonical decode/re-encode sequence; context ids never reconstructed by
  re-tokenizing; wrong prediction indexing fails **loudly** (diverges); runtime
  verify failure degrades to plain without corrupting output.
- **Duplicate surface / non-canonical** (`results/spec_phase2/correctness.json`,
  `token_id_all_equal: true`): token mode reproduces the exact ids, and a
  text-surface reconstruction of the *same output* would re-tokenize to a
  *different* id sequence — the concrete proof that surface identity is
  insufficient and id identity is not.
- **Text proposer → token-id draft is conservative**: a proposed string is only
  used when re-tokenizing `context_text + draft_text` reproduces the exact
  authoritative context ids as a prefix; otherwise it is rejected
  (`context_prefix_mismatch`) and a plain burst makes exact progress.

## 6. Correctness results — bytes, Unicode and seams

- **Bytes** follow from ids: every token-ID row asserts `text_bytes` equality too.
- **Unicode / seam (token mode)** — id + byte identity across ASCII, `café`,
  `Shqipëria është e bukur`, emoji, curly quotes / em-dashes, combining marks,
  leading/trailing spaces, multiple newlines, repeated punctuation, and a merge
  token spanning the context/draft boundary (`test_unicode_and_seam_token_identity`,
  `test_token_merge_across_context_draft_boundary_stays_exact`).
- **Unicode / seam (text mode)** — byte identity across the same strings against a
  **code-point** endpoint (`test_unicode_and_seam_byte_identity_text_mode`); the
  **byte-offset** endpoint over multibyte text is **rejected**, not mis-verified.

## 7. Negative results (kept visible)

- **`llama-cpp-python` is not installed and no real/trained GGUF is reachable
  here.** Token-ID equivalence on a real engine is therefore **not** demonstrated
  in this environment — only its deterministic-fixture equivalence is. This is a
  reported gap, not worked around.
- **Text-surface mode still cannot prove token-id identity** by construction. It
  is retained only as a conditional, experimental compatibility lane.
- **Native `llama.cpp` `llama-server`** still does not expose prompt-position
  logprobs (Phase 1) → text mode remains unusable against it.
- **The pathological random fixture** re-tokenizes non-canonically; for text mode
  a safe rejection/fallback is the correct result, and for token mode a burst
  fallback keeps output exact — neither is a workload result.
- **No wall-clock speedup.** On this overhead-dominated CPU rig the timed sim
  benchmark reports a **slowdown** (~0.11×, i.e. spec's 7 round-trips vs 1 make it
  ~8× slower), consistent with Phase 1.

## 8. Benchmark methodology (corrected) — but no speed claim

`sclab.spec.bench` now **gates speed on correctness** and, only after byte
identity passes, times with: validated timed calls (a fast error is never a fast
success), alternating baseline/spec order, cold-prefix / warm-prefix /
fully-cached regimes (approximated by prompt construction — the engine cache is
not resettable via the API, and this is labelled), **median with min/max**
(not best-of), HTTP request count reported **separately** from wall time, and a
`prefix_cache_observed` flag. Raw machine-readable output is saved via
`save_bench_results`. See `results/spec_phase2/benchmark.json`: on the
**overhead-dominated** synthetic sim (no modeled decode cost, so the sim's own
per-request cost dominates) speculation makes **7 HTTP requests vs the baseline's
1** and is correspondingly **~8× slower** (`speedup_median ≈ 0.11`,
`prefix_cache_observed: false`) — the honest CPU-rig result, not a speedup. A
modeled decode cost could manufacture a "win" here, which would prove nothing.
**No number here is a trained-model or bandwidth-bound-decoder result.**

## 10. What remains unproven

1. **Token-ID equivalence on a real engine** (`llama-cpp-python` in-process) and
   on a **trained** model, across JSON / tool schema / code / quoted RAG / prose /
   Unicode — set up as opt-in tests, **not run** (no engine / no weights here).
2. **Any speedup**, which needs a memory-bandwidth-bound decoder (large model /
   GPU / cheap incremental scoring) — still unavailable.
3. **Other engines** (vLLM `prompt_logprobs`, SGLang, TGI) — each needs its own
   probed adapter; none tested. The token-ID adapter is **not** universal.

   bandwidth-bound decoder and report it **beside** the correctness result.
